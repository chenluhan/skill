# prototype-to-prd-pack 设计说明

## 结论

建议创建一个名为 `prototype-to-prd-pack` 的 skill。

它的定位不是“把原型描述写成一篇 PRD”，而是：

- 从原型、截图、链接、补充说明中重建完整产品规格
- 输出一套同时适合研发阅读和 AI 消费的 PRD 交付包
- 稳定生成 Mermaid 流程图并导出 PDF
- 对缺失信息保持克制，用 `明确事实 / 稳定推断 / 待确认项` 三类标注控制风险

第一版不做高保真设计生成，不做技术架构设计，不做严格 BPMN 建模。

## 目标

这个 skill 必须解决四个核心问题：

1. 输入很乱也能归一
2. 推断不胡编
3. 图能稳定产出
4. PDF 能闭环导出

在这四点之上，还必须保证 PRD 本身质量足够高。高质量的定义不是“文案像 PRD”，而是：

- 研发读完后知道要实现什么、哪些状态不能漏
- AI 读完后能继续拆任务、写测试、生成实现草案
- 评审时能主动暴露缺口，而不是用话术掩盖不确定性

## 用户问题

基于原型生成 PRD 的常见问题，不在于“不会写文档”，而在于原型只展示了用户能看到的主路径，没有展示完整规则面。直接从原型写 PRD，通常会出现以下问题：

- 只复述页面视觉结构，没有重建产品任务和业务规则
- 主路径存在，异常、边界、权限、空态、失败态缺失
- 流程图只是装饰，不能反映角色边界与异常回路
- 文档对人类可读，但对 AI 不够结构化
- 模型会把猜测伪装成事实，导致研发和 AI 后续继续误解

这个 skill 的本质工作，不是写作，而是重建规格。

## 推荐定位

三种可能路线中，推荐采用 `PRD 交付包生成器` 路线。

### 不推荐：轻量转换器

- 输入原型后直接输出一篇 PRD
- 好处是快
- 问题是质量上限低，容易退化成页面描述

### 推荐：PRD 交付包生成器

- 先抽取页面、流程、角色、状态线索
- 再补检规则、异常、权限和边界
- 最后输出多文件交付包、Mermaid 图源和 PDF

这是第一版最均衡的方案。

### 第一版不做：产品重建设计师

- 这条路线会进一步做信息架构重设、方案优化、优先级裁剪
- 上限高，但边界太大
- 第一版会失焦，不利于把“原型转规格”的主能力做稳

## 输入策略

默认采用 `混合输入`。

支持以下输入组合：

- 原型链接
- 页面截图
- 原型插件导出的页面说明
- 用户补充的业务背景或约束描述

Skill 不应要求输入干净整齐。它应先把输入统一归一化，再进入分析。

### 输入处理原则

- 对可见内容先提取事实，不做脑补
- 对上下文能稳定推断的内容，标记为 `稳定推断`
- 无法稳定确定的内容，进入 `待确认项`

这个原则必须写进 skill 的硬约束，不能只作为提示。

## 范围与非目标

### 范围

这个 skill 负责：

- 页面清单抽取
- 页面结构抽取
- 页面间跳转关系梳理
- 用户主任务流程重建
- 状态、异常、权限、规则补检
- Mermaid 流程图与时序图生成
- 多文件 PRD pack 输出
- PDF 汇总导出

### 非目标

这个 skill 不负责：

- 生成高保真 UI 视觉稿
- 替代产品经理做业务战略判断
- 对严重缺失的输入强行补全成完整事实
- 默认输出技术架构、数据库设计、接口设计
- 严格 BPMN 级别的流程建模

## 输出包结构

输出不应是散落文件，而应是一个可编辑交付包加一个可分发 PDF。

建议默认目录结构为：

```text
<project-name>-prd-pack/
├── 00-overview.md
├── 01-prd.md
├── 02-pages.md
├── 03-flows.md
├── 04-ai-spec.md
├── 05-open-questions.md
├── diagrams/
│   ├── flow-main.mmd
│   ├── flow-exceptions.mmd
│   ├── sequence-*.mmd
│   └── rendered/
│       ├── flow-main.svg
│       ├── flow-exceptions.svg
│       └── sequence-*.svg
├── assets/
│   ├── page-index.json
│   ├── screen-map.json
│   └── evidence.json
└── export/
    └── prd-pack.pdf
```

### 00-overview.md

用于让人类在 3 分钟内读懂全貌，包含：

- 产品目标
- 交付范围
- 术语与角色
- 页面列表
- 主流程摘要
- 风险摘要

### 01-prd.md

作为传统 PRD 主文档，包含：

- 背景
- 目标
- 范围
- 用户故事
- 功能说明
- 业务规则
- 非目标

### 02-pages.md

作为逐页规格文档。每个页面必须固定包含：

- 页面目标
- 入口
- 出口
- 关键信息区块
- 用户动作
- 系统反馈
- 状态集合
- 异常处理
- 权限限制
- 埋点建议

### 03-flows.md

作为所有 Mermaid 图的可读总览。每张图都必须配套：

- 图用途
- 适用范围
- 关键节点解释
- 已知缺口

### 04-ai-spec.md

作为 AI 优化的结构化规格。其职责是让 AI 能直接消费，而不是复述 PRD 正文。它必须使用稳定字段和固定层次。

### 05-open-questions.md

用于集中承接所有待确认项。每条问题必须包含：

- 来源页面或流程
- 问题描述
- 对交付的影响
- 建议确认角色
- 当前默认假设

### 机器层资产

`assets/page-index.json` 和 `assets/evidence.json` 是必须项。

- `page-index.json` 记录页面 ID、页面名称、关联流程、来源证据
- `evidence.json` 记录每条关键结论的证据类型：`明确事实 / 稳定推断 / 待确认项`

这两层资产用于约束文档稳定性，并为后续 AI 继续消费提供基础。

## Mermaid 与 PDF 策略

这个 skill 必须采用 `Mermaid-first` 的图形系统。

原因：

- Mermaid 源码可编辑、可版本化、可被 AI 再消费
- 对评审展示来说，源码、图像、文档说明三层可以保持一致
- 比只输出静态图片更适合持续迭代

### 图类型策略

默认只保留两大类图：

- 泳道流程图
- 时序图

### 泳道流程图

用于表达跨角色、跨系统主流程。

重点表现：

- 正常主链路
- 关键分支
- 失败回路
- 人工介入点

推荐 lane 顺序：

- 用户
- 客户端 / App
- 服务端
- 运营 / 审核 / 人工
- AI 能力

按具体场景可裁剪，不应机械保留全部 lane。

### 时序图

用于表达关键交互时序。

重点表现：

- 请求与响应
- 异步返回
- 重试
- 超时
- 降级
- 人工接管

AI 场景中，时序图比普通流程图更适合表达不确定返回和失败兜底。

### Mermaid 的真实边界

这个 skill 需要明确告知使用者：

- `sequenceDiagram` 很适合时序表达，应作为强默认
- Mermaid 没有严格 BPMN 级泳道建模能力
- 第一版泳道图应使用 `flowchart + subgraph` 的约定化泳道模式
- 当泳道图表达不清时，允许退化为按角色分组的 flowchart，并用文字补充边界说明

这会牺牲严格建模能力，但能换来更高的生成稳定性和更低的维护成本。

### 流程图最小交付要求

每条主流程默认至少输出：

- 1 张主泳道图
- 1 张关键异常或分支泳道图（异常复杂时）
- 1 到 3 张关键时序图

### PDF 规则

最终必须生成 `export/prd-pack.pdf`。

PDF 不应是所有 Markdown 的机械拼接，而应按评审阅读顺序重组：

1. 封面与摘要
2. 主流程总览
3. 主 PRD
4. 逐页规格
5. 流程图与时序图
6. 待确认问题清单

如果本地依赖缺失导致无法导出 PDF，skill 必须：

- 明确说明卡在哪一步
- 保留所有中间产物
- 不得假装 PDF 已完成

## 质量引擎

PRD 质量不是附加项，而是这个 skill 的核心。

建议将整个 skill 的方法论写成：`analysis-first, writing-last`。

也就是先重建规格，再把规格编译成文档。

### Pass 1：事实抽取

只记录看得见的内容：

- 页面
- 模块
- 按钮
- 文案
- 跳转线索
- 角色线索
- 状态线索

这一层不允许脑补业务规则。

### Pass 2：产品重建

从事实反推出：

- 目标用户
- 核心任务
- 页面清单
- 主流程
- 分支流程
- 业务对象
- 状态与异常面

这一层先形成规格判断，不直接输出最终文档。

### Pass 3：完整性补检

基于固定清单补查缺口，重点检查：

- 页面层：入口、出口、信息层级、交互反馈、空态、加载态、失败态、权限态
- 流程层：前置条件、主路径、异常分支、中断恢复、完成条件
- 规则层：边界值、频控、幂等、资格、角色差异、人工介入点
- AI 层：不确定性、超时、降级、重试、人工接管

### Pass 4：多文档生成

把已经重建好的规格分别编译成：

- 人类版 PRD
- 逐页规格
- Mermaid 流程图
- AI-ready spec
- 待确认问题清单

### Pass 5：自检与降噪

生成后必须自检：

- 有没有把猜测写成事实
- 有没有 generic PM 套话
- 页面规格是否缺栏
- 流程图和正文是否矛盾
- AI spec 与 PRD 是否字段错位
- 待确认项是否遗漏或散落

## 硬性质量规则

以下规则必须进入 skill 的硬约束：

### 1. 每条关键结论都要有来源

来源只能是三类：

- `明确事实`
- `稳定推断`
- `待确认项`

不允许第四类“模糊常识脑补”。

### 2. 每个页面必须写完整

最少包含：

- 页面目标
- 入口
- 出口
- 关键信息区块
- 用户动作
- 系统反馈
- 状态集合
- 异常处理
- 权限限制
- 埋点建议

### 3. 每条流程必须闭环

至少说明：

- 谁触发
- 在什么前提下触发
- 主路径怎么走
- 哪些地方会失败
- 失败后怎么办
- 什么算完成

### 4. 每条业务规则必须可执行

不能停留在“系统应支持”这种空话，必须明确：

- 触发条件
- 限制条件
- 系统行为
- 用户可见反馈
- 异常兜底

### 5. 待确认项必须集中管理

一旦原型无法支撑判断，就进入 `05-open-questions.md`。不能在正文里一边默认、一边不声明。

## Skill 内部资源设计

这个 skill 必须自带脚本和模板，不能只依赖 SKILL.md 文本说明。

建议目录结构：

```text
prototype-to-prd-pack/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── scripts/
│   ├── normalize_inputs.py
│   ├── build_pack.py
│   ├── render_mermaid.sh
│   └── export_pdf.sh
├── references/
│   ├── page-spec-template.md
│   ├── prd-template.md
│   ├── ai-spec-template.md
│   ├── flow-rules.md
│   ├── inference-rules.md
│   └── open-questions-template.md
└── assets/
    └── examples/
```

### scripts/normalize_inputs.py

职责：

- 统一处理链接、截图、文字说明等混合输入
- 输出中间结构

建议输出内容：

- 页面列表
- 页面证据
- 可见流程
- 角色线索
- 状态线索
- 缺失项

### scripts/build_pack.py

职责：

- 根据中间结构生成整个 PRD pack
- 同时生成 `assets/page-index.json` 与 `assets/evidence.json`

### scripts/render_mermaid.sh

职责：

- 批量把 `.mmd` 渲染成 `.svg`
- 固定 Mermaid CLI 调用方式和输出路径

### scripts/export_pdf.sh

职责：

- 将 Markdown、SVG 和导出顺序组装成 PDF
- 第一版优先使用本地已有链路，例如 `pandoc` 或浏览器打印

### references/page-spec-template.md

定义逐页规格固定栏目，避免页面规格输出不一致。

### references/prd-template.md

定义主 PRD 固定段落，避免风格漂移。

### references/ai-spec-template.md

定义 AI 结构化规格字段，确保后续 AI 消费稳定。

### references/flow-rules.md

定义 Mermaid 写法，包括：

- 泳道顺序
- 节点命名规范
- 异常分支画法
- 时序图的超时、重试、兜底表达

### references/inference-rules.md

定义什么可以推断，什么必须打成待确认项。

这是防止 skill 胡编的核心闸门。

### references/open-questions-template.md

统一待确认项表达方式，避免输出变成随手备注。

## SKILL.md 核心工作流

`SKILL.md` 不应该教另一个 Codex “怎么把话写漂亮”，而应该强约束它按固定顺序思考和执行。

建议核心工作流如下：

### Step 1：判断输入完整度

- 识别输入类型：原型链接、截图、文字说明、混合输入
- 如果信息足够，直接进入分析
- 如果缺口明显，先整理最小待确认清单，不立刻写 PRD

### Step 2：归一化输入

- 调用 `scripts/normalize_inputs.py`
- 输出统一中间结构

### Step 3：重建产品规格

先还原：

- 目标用户
- 核心任务
- 页面清单
- 主流程
- 分支流程
- 业务对象
- 状态与异常面

### Step 4：执行完整性补检

读取以下 references：

- `references/inference-rules.md`
- `references/flow-rules.md`

并执行页面、流程、规则、异常、AI 专项补检。

### Step 5：生成交付包

- 调用 `scripts/build_pack.py`
- 输出全部 Markdown、JSON、Mermaid 源文件

### Step 6：渲染 Mermaid

- 调用 `scripts/render_mermaid.sh`
- 输出 SVG 图

### Step 7：导出 PDF

- 调用 `scripts/export_pdf.sh`
- 输出 `export/prd-pack.pdf`

### Step 8：最终自检

对照模板和规则检查：

- 图文是否一致
- 推断项是否明确标注
- 待确认项是否汇总完整
- PDF 是否包含关键图与核心文档

## 触发描述建议

这个 skill 的前言 description 应明确覆盖以下触发场景：

- 基于原型生成 PRD
- 根据截图或页面草图梳理产品规格
- 从 AI Studio、Figma 或原型插件输出完整 PRD 包
- 补齐页面交互、异常情况、状态矩阵、流程图、线框说明
- 生成既适合研发阅读、也适合 AI 消费的产品规格包

它不应被写成一个泛化的“任何 PRD 都能写”的 skill，而应明确聚焦在“从原型或页面设计还原规格”。

## 成功标准

这个 skill 的成功标准至少包括：

1. 输入混乱时仍能生成稳定结构
2. 输出文档显式区分事实、推断和待确认项
3. 页面规格能系统覆盖状态、异常、权限和规则
4. 流程图不是装饰，而能表达角色边界、异常回路和关键时序
5. PDF 导出形成完整闭环
6. AI 能直接消费 `04-ai-spec.md`、Mermaid 图源和 JSON 资产继续工作

## 实施建议

实现顺序建议为：

1. 先初始化 skill 目录
2. 先写 references 模板和规则文件
3. 再写 `SKILL.md`
4. 再补 `normalize_inputs.py`、`build_pack.py`
5. 最后补 Mermaid 渲染和 PDF 导出脚本

原因是这个 skill 的核心先是规则和模板稳定，然后才是自动化脚本闭环。顺序反过来，脚本会先固化错误结构。

## 当前设计结论

推荐第一版按以下方案落地：

- skill 名称：`prototype-to-prd-pack`
- 默认输入：混合输入
- 默认输出：PRD 交付包 + Mermaid 图源 + PDF
- 默认图系统：泳道流程图 + 时序图
- 默认方法：`analysis-first, writing-last`
- 默认风险控制：`明确事实 / 稳定推断 / 待确认项`

这套方案足够聚焦，也足够强，适合作为第一版正式实现。
